# FIPS Base Images

Minimal, FIPS-hardened OCI container base images built with
[apko](https://github.com/chainguard-dev/apko) and
[Wolfi](https://github.com/wolfi-dev/os) open-source packages.

## Images

| Image | Definition | Description |
|---|---|---|
| `fips-base` | `images/base/image.yaml` | Wolfi baseOS with OpenSSL FIPS provider + jitterentropy |
| `fips-java21` | `images/java21/image.yaml` | Java 21 JRE on top of `fips-base`, configured for FIPS via SunPKCS11 |

Both images target `x86_64` and `aarch64`.

## FIPS Architecture

### Kernel-Independent Design

Both images include `jitterentropy-library`, a NIST SP 800-90B validated
userspace entropy source. This makes the OpenSSL FIPS provider
(CMVP certificate [#4282](https://csrc.nist.gov/projects/cryptographic-module-validation-program/certificate/4282))
fully self-contained — the host kernel does **not** need to be in FIPS mode.
Images run on GKE (COS), AWS Bottlerocket, Azure Linux, and standard kernels.

### Base image (`fips-base`)

| Package | Role |
|---|---|
| `openssl` | OpenSSL 3.4+ with built-in FIPS provider module and FIPS-only algorithm patches |
| `jitterentropy-library` | SP 800-90B userspace entropy source (kernel-independent) |
| `ca-certificates-bundle` | Trusted TLS root certificates |
| `wolfi-base` | BusyBox + musl libc minimal userspace |

### Java 21 image (`fips-java21`)

Java FIPS is achieved via the **SunPKCS11** provider backed by the OpenSSL FIPS
provider module. All JCA/JCE operations are routed through the FIPS cryptographic
boundary.

| Package | Role |
|---|---|
| `openjdk-21-jre` | OpenJDK 21 Java Runtime Environment |
| `openjdk-21-default-jvm` | Sets OpenJDK 21 as the default JVM |
| *(all base packages)* | OpenSSL FIPS + jitterentropy, as above |

Runtime configuration files are provided in `images/java21/`:

- `pkcs11-fips.cfg` — SunPKCS11 provider configuration pointing to `fips.so`
- `fips.security` — Java security properties override enabling FIPS providers and restricting non-FIPS algorithms

#### Activating FIPS in Java

Copy the config files into the image and set the environment variable:

```dockerfile
COPY images/java21/pkcs11-fips.cfg  /etc/java/pkcs11-fips.cfg
COPY images/java21/fips.security    /etc/java/fips.security
ENV  JAVA_TOOL_OPTIONS="-Djava.security.properties=/etc/java/fips.security"
```

Or pass it at runtime:

```bash
java -Djava.security.properties=/etc/java/fips.security -jar myapp.jar
```

> **Commercial alternative:** Chainguard's commercial Wolfi packages include
> `bouncycastle-fips` (CMVP [#4943](https://csrc.nist.gov/projects/cryptographic-module-validation-program/certificate/4943)),
> which provides a fully Java-native FIPS 140-3 stack with the BC-RNG-JENT
> entropy provider — eliminating the PKCS11 bridge entirely.

## Prerequisites

Install [apko](https://github.com/chainguard-dev/apko):

```bash
# via Homebrew
brew install apko

# via Go
go install chainguard.dev/apko@latest
```

## Building

```bash
# Build both images
make all

# Build individually
make base
make java21

# Load into local Docker daemon
make load-base
make load-java21

# Clean build artefacts
make clean
```

Build outputs:

- `fips-base.tar` / `fips-java21.tar` — OCI image tarballs (multi-arch)
- `fips-base.sbom.json` / `fips-java21.sbom.json` — SBOMs generated by apko

Override the registry or apko binary:

```bash
make all REGISTRY=gcr.io/my-project APKO=/usr/local/bin/apko
```

## Verifying FIPS

```bash
# Load base image
docker load < fips-base.tar

# Confirm OpenSSL is in FIPS mode (MD5 should be blocked)
docker run --rm fips-base:latest sh -c \
  'openssl md5 /dev/null 2>&1 && echo "FAIL: MD5 allowed" || echo "OK: MD5 blocked (FIPS active)"'

# Confirm OpenSSL version and FIPS provider
docker run --rm fips-base:latest openssl version -a
docker run --rm fips-base:latest openssl list -providers

# Load Java image
docker load < fips-java21.tar

# Check Java version
docker run --rm fips-java21:latest java -version
```
